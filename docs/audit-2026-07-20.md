## Findings and dispositions

### P0 — data loss / privacy / account isolation

1. ✅ **New account inherited device data.** `decidePull` returns `push` when no remote doc
   exists (`cloudSync.ts`), which uploaded the device's whole local state into a brand-new
   account. Fixed by account-scoped namespaces (`profileStorageKey`, `ProfileRepository`,
   `switchActiveProfile`): a new account's namespace is empty, so the seed push uploads nothing.
2. ✅ **Sign-out left previous account's data visible.** Auth listener now switches to the
   anonymous profile on sign-out; account data is unloaded immediately.
3. ✅ **Pending debounced write dropped on sign-out.** `signOut()` cancelled `pushTimer`
   (up to 1.5 s of logged sets). Now flushes via `pushNow()` when dirty before signing out.
4. ✅ **Photos crossed account boundaries.** `mergeRemote` kept `local.photos` while adopting
   another account's remote data. Photos now live inside each profile's namespace.
5. ✅ **Sync metadata not identity-scoped.** Single `bodyos.sync.meta.v1` key; now per-UID.
6. ✅ **Silent persistence failure.** `LocalStorageRepository.save()` swallowed quota/privacy
   errors ("data stays in memory for this session"). `save()` now reports success; the store
   tracks `storageFailing`; a persistent `role=alert` banner shows in AppShell and Gym Mode.
7. ✅ **No Firebase rules in version control.** `firestore.rules` (hardened with shape/version/
   server-timestamp validation), `firebase.json`, `.firebaserc` added. **Console action
   required:** deploy rules so live matches VCS.
8. ✅ **Normalized per-entity sync is now the LIVE path**, replacing whole-blob LWW.
   `lib/syncDiff.ts` (entity diff), `store/syncQueue.ts` (IndexedDB mutation queue +
   per-entity rev baseline), `store/syncEngine.ts` (`pushMutation`/`drainQueue`/`pullRemote`
   with optimistic-concurrency conflict detection + a `conflicts/` shelf so a losing edit is
   never silently discarded — a concurrent delete never forces through a concurrent edit
   either), `store/syncMigration.ts` (idempotent blob → normalized fan-out on first sign-in,
   legacy blob kept in place as a rollback window), and `firestore.rules` (owner-only,
   shape-validated normalized collections, additive alongside the legacy doc) are all wired
   into `cloudSync.ts` as the actual sign-in/write/sign-out/delete-account path. 34 emulator
   tests + 27 unit tests, including two integration tests that exercise the real module
   wiring end-to-end (sign-in → migrate → pull → catch-up → drain → Firestore; account
   deletion purging every collection, not just the legacy blob). Three real bugs caught by
   that suite before touching anything live: a premature rev-baseline bump that misfired
   every fresh entity as a conflict, a cross-file emulator test race, and (would have been)
   an incomplete account deletion once real data lived in the new collections.
   **Remaining, deliberately deferred:** a conflict-shelf UI surface in Settings so a shelved
   conflict is visible without opening the Firebase console — low-frequency (needs genuine
   concurrent edits from two devices) and non-destructive, but a real gap for a "you'll
   never lose data and always know it" promise — and the "tombstone should generally beat a
   concurrent edit" refinement noted in `syncPull.ts` (currently the edit wins, which is
   safe — nothing is lost — but not what the design spec intends). **Console action still
   required for this to be enforced live:** deploy `firestore.rules` (item #7) — the new
   collections' rules only take effect once deployed.
9. ✅ **Firebase emulator suite is wired up and used** (`npm run test:emulator`; CI runs it).
   Rules, push/pull reconciliation, migration, AND the real `cloudSync.ts` module wiring
   (sign-in → migrate → pull → catch-up → drain → Firestore; account deletion purging every
   collection) are all exercised against a REAL emulated Firestore + the actual
   `firestore.rules` file — not mocks.
10. ✅ **Photos as data-URLs inside AppData** — payloads now live in a profile-scoped
    IndexedDB store (`photoStore.ts`); the persisted document carries metadata only and a
    startup migration shrinks legacy installs. Workout saves can no longer be blocked by
    the photo library.
11. ✅ **Account deletion** — `deleteCloudData` + `deleteAccount` (reauth-aware, partial-failure
    honest) + `localAccountCleanup`; wired into the Account panel. Firebase glue still needs
    emulator coverage.
12. 🔴 **Dev and prod share one Firebase project** (`bodyos-e7372`).

### P1 — blocks reliable daily use

13. ✅ **User zoom disabled** (`maximum-scale=1, user-scalable=no` in `index.html`) — removed.
14. ✅ **No PR CI.** `ci.yml` added: typecheck + tests + build + `git diff --check` on every PR
    and push to main (audit advisory-only until triaged). Branch protection: enable
    "Require status checks → CI / validate" in repo settings (console action).
15. ✅ **Backup validation deepened** — nested entities validated with exact failure locations,
    size guard, hostile-payload tests; restores snapshot the previous data with an
    "Undo last restore" path. (Merge-import remains replace-only, stated in the UI.)
16. ✅ **Sheet accessibility** — focus trap/initial focus/restore, `aria-labelledby`, scroll
    lock, tested. (Nested-sheet stacking + true `inert` + on-device SR pass remain.)
17. ✅ **First-session honesty** — no invented default; explicit "set your starting weight"
    state in Gym Mode; PlateBar hidden until a real load exists.
18. ✅ **Set metric model v1** — `ExerciseMetric` ('load-reps' | 'bodyweight-reps' | 'duration');
    plank is now a timed hold (30–60 s), pull-ups/dips/push-ups etc. are bodyweight+added-load;
    Gym Mode steppers/labels, objectives, set ledger, beat lines, announcements and the
    progression engine all judge by the right unit (duration recs never mention kg).
    (Assisted-bodyweight, distance and carries remain for v2.)
19. ✅ **Historical editing (v1)** — `/history/:id` reviews and corrects past sets, renames and
    deletes sessions; PRs + streaks recompute from source sessions (`lib/recompute.ts`),
    stale stored recommendations dropped. (Time editing + add/remove exercises remain.)
20. 🟠 **Partial workouts**: `completeSession` treats any finish as `completed`; no
    partial/abandoned distinction in summaries or progression gating (spec §2.7).
