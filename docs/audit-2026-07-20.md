# BodyOS production-readiness audit — 2026-07-20

Baseline (before this audit's fixes): 124 tests green, `tsc` clean, build green,
dist ≈ 2.0 MB (largest gz chunks: firebase 175 kB, recharts-shared 107 kB, app 88 kB),
`npm audit` reports advisories (untriaged), CI runs only on push to `main`.

Status legend: ✅ fixed in this pass · 🔴 open P0 · 🟠 open P1 · 🟡 open P2 · ⚪ P3.

## Confirmed working (verified, not assumed)

- Fresh installs start empty; demo data is opt-in and clearable (`loadOrCreate`,
  `createEmptyData`, `loadDemo`, `clearHistory`; `repository.test.ts`, `accountProfiles.test.ts`,
  scripted E2E flows 2026-07-20).
- Gym Mode core loop (log → advance → undo → edit → finish) — `GymMode.test.tsx`, `gymFlow.test.ts`.

- Backup export/import with validated `parseBackup` (shallow — see 🟠 below).
- Offline-first: no network dependency in the workout path; SW precaches the shell.

## Findings and dispositions

### P0 — data loss / privacy / account isolation

1. ✅ **New account inherited device data.** `decidePull` returns `push` when no remote doc
   exists (`cloudSync.ts`), which uploaded the device's whole local state into a brand-new
   account. Fixed by account-scoped namespaces (`profileStorageKey`, `ProfileRepository`,
   `switchActiveProfile`): a new account's namespace is empty, so the seed push uploads nothing.
2. ✅ **Sign-out left previous account's data visible.** Auth listener now switches to the
   anonymous profile on sign-out; account data is unloaded immediately.
3. ✅ **Pending debounced write dropped on sign-out.** `signOut()` cancelled `pushTimer`
   (up to 1.5 s of logged sets). Now flushes via `pushNow()` when dirty before signing out.
4. ✅ **Photos crossed account boundaries.** `mergeRemote` kept `local.photos` while adopting
   another account's remote data. Photos now live inside each profile's namespace.
5. ✅ **Sync metadata not identity-scoped.** Single `bodyos.sync.meta.v1` key; now per-UID.
6. ✅ **Silent persistence failure.** `LocalStorageRepository.save()` swallowed quota/privacy
   errors ("data stays in memory for this session"). `save()` now reports success; the store
   tracks `storageFailing`; a persistent `role=alert` banner shows in AppShell and Gym Mode.
7. ✅ **No Firebase rules in version control.** `firestore.rules` (hardened with shape/version/
   server-timestamp validation), `firebase.json`, `.firebaserc` added. **Console action
   required:** deploy rules so live matches VCS.
8. 🔴 **Whole-blob last-write-wins sync remains.** One Firestore doc per user; every edit
   uploads everything; two-device concurrent edits lose one side silently; doc-size ceiling
   (1 MiB) will eventually bite. Direction: normalized per-entity collections
   (`users/{uid}/sessions/{id}` …) with tombstones + offline mutation queue (spec §0.2/0.3).
   Largest remaining P0; needs its own design slice + emulator tests + blob migration.
9. 🔴 **No Firebase emulator tests** of auth lifecycle, rules, or reconcile glue. The pure
   parts (`decidePull`, merge, profiles) are unit-tested; the wiring is not.
10. 🔴 **Photos as data-URLs inside AppData** — a few photos can exhaust localStorage and
    (until this pass, silently) block workout saves. Now loudly surfaced, but the fix is
    moving blobs to IndexedDB with metadata separation (spec §0.5).
11. 🔴 **No complete account deletion** (Firestore doc + auth identity + local namespaces).
12. 🔴 **Dev and prod share one Firebase project** (`bodyos-e7372`).

### P1 — blocks reliable daily use

13. ✅ **User zoom disabled** (`maximum-scale=1, user-scalable=no` in `index.html`) — removed.
14. ✅ **No PR CI.** `ci.yml` added: typecheck + tests + build + `git diff --check` on every PR
    and push to main (audit advisory-only until triaged). Branch protection: enable
    "Require status checks → CI / validate" in repo settings (console action).
15. 🟠 **Backup validation is shallow** (`parseBackup` checks top-level shape only; nested
    sets/timestamps/references unvalidated; restore is destructive with no pre-backup,
    no merge option, no schema-version migration path).
16. 🟠 **Sheet/dialog accessibility**: `Sheet.tsx` has `role=dialog`+Escape but no focus trap,
    no initial focus, no focus restore, no `aria-labelledby`, no body scroll lock.
17. 🟠 **First-session honesty**: `prefillFor` defaults to `startWeightKg ?? 20` kg
    (`history.ts:92`) — an arbitrary load presented like a recommendation.
18. 🟠 **Set model too narrow**: `weightKg + reps` only (`types/index.ts`); planks, carries,
    pull-ups/assisted, timed holds don't fit (spec §2.4). Type-level change rippling through
    Gym Mode, PRs, progression, analytics, backup.
19. 🟠 **Historical edit surface incomplete**: sets are editable in-session; completed sessions
    have no edit path (name/time/exercises/sets), and PR/analytics recomputation after edits
    is untested (spec §2.6; `detectPersonalRecords` runs at completion only).
20. 🟠 **Partial workouts**: `completeSession` treats any finish as `completed`; no
    partial/abandoned distinction in summaries or progression gating (spec §2.7).

### P2 — needed for a strong beta

21. 🟡 **Equipment ignored by starter routines** — onboarding collects `equipment` but
    `routines.ts` never consults it; a no-barbell user gets barbell-centric splits.
22. 🟡 **Onboarding copy vs reality**: goal/experience are stored; progression currently uses
    neither ("This sets sensible starting loads" is not implemented). Fix copy or implement.
23. 🟡 **Progression engine sequence-level verification**: unit tests exist
    (`progression.test`, `plan`, `prstats`), but stall counting across edited/deload/mixed
    sequences lacks table-driven multi-session tests (spec §2.1 — items to reproduce, not assumed).
24. 🟡 **Sparse-data confidence**: trends render from 2 points; no minimum-sample gating.
25. 🟡 **PWA update safety**: `autoUpdate` SW can reload during an active workout; no
    update-ready UI, no version display; single SVG icon (no PNG set for iOS).
26. 🟡 **No lint/format/coverage/E2E scripts** (Playwright harness exists as scripts, not CI).
27. 🟡 **No privacy documentation** (photos, measurements, sync, deletion, Firebase processor).
28. 🟡 **Lighthouse not measured** in any pipeline.
